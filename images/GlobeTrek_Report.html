1. Introduction

This report documents the planning, design, implementation, and testing of a full-stack web application developed for GlobeTrek Adventures, a newly established travel and tourism company based in Negombo, Sri Lanka. The system was developed as part of the CSE5009 Web Application Development module assessment (WRIT1), encompassing both academic research and practical software engineering.

Travel and tourism management is a complex, multifaceted domain that requires careful co-ordination of tour packages, customer bookings, staff operations, and administrative oversight. GlobeTrek Adventures required a modern, interactive web-based platform that allows customers to browse and book curated Sri Lankan travel packages, enables travel agency staff to manage bookings and respond to inquiries, and provides administrators with full control over system users and operations.

The platform was designed and built as a full-stack PERN-influenced application, using React 18 on the frontend and Node.js / Express.js with a PostgreSQL relational database on the backend. The system implements three distinct user roles — Customer, Staff, and Administrator — each with carefully scoped access controls enforced by JSON Web Token (JWT) authentication.

1.1 Project Scope and Objectives

  • Provide an attractive, mobile-responsive public interface showcasing Sri Lanka tour packages.
  • Allow customers to register, log in, search packages, make bookings, and track booking status.
  • Enable staff to manage bookings, update tour package details, and respond to customer inquiries.
  • Provide administrators with user management, reporting, and full system oversight capabilities.
  • Implement robust input validation, authentication, and role-based access control (RBAC).
  • Follow modern web development standards including RESTful API design and component-based UI architecture.

1.2 Technology Stack Summary

The platforms selected share a similar target audience — travellers seeking guided tour packages — and represent both global and regional market leaders.

Systems Selected for Comparison

  • System A — TripAdvisor.com: A globally dominant travel platform offering user reviews, hotel and tour bookings, and destination guides. It serves millions of users worldwide and is renowned for its community-driven content model.

  • GlobeTrek Adventures (our system): A purpose-built Sri Lanka–focused travel booking platform catering to niche adventure and cultural tourism packages.

Critical Analysis and Design Decisions Derived

The comparative analysis revealed several important insights that directly shaped the design of GlobeTrek Adventures:

  1. Simplicity over feature bloat: TripAdvisor's mega-menu navigation, while powerful, introduces cognitive overload for first-time users. GlobeTrek adopted a flat four-item navigation to eliminate friction for the target tourist demographic.
  2. Role-adaptive UI: MakeMyTrip lacks any staff/admin interface. GlobeTrek's inline role-based dashboard — where the same URL renders different tabs depending on the user's role — was identified as a competitive differentiator that reduces operational complexity for the travel agency.
  3. Performance prioritisation: Both competitor platforms suffer from slow load times due to third-party ad scripts and heavy promotional content. GlobeTrek's Vite-built, CSS Modules architecture achieves significantly faster load times with a 323 kB total JS bundle.
  4. Inline booking: MakeMyTrip's booking process requires multiple page transitions. GlobeTrek's booking widget is embedded directly on the tour detail page, reducing the booking funnel to a single page interaction.
  5. Inquiry management loop: Neither competitor provides a visible, trackable inquiry thread for customers. GlobeTrek's inquiry system provides end-to-end status visibility (New → Read → Responded → Closed) for both the customer (via their dashboard) and staff (via the admin dashboard).
Summary Verdict: GlobeTrek Adventures matches or exceeds both comparison systems in Navigation, Performance, and Role-Based Access. The primary area for future enhancement is payment gateway integration and AI-driven recommendations, which remain the key differentiators of the more mature platforms.

2.2.3 Typography

A single typeface — Poppins (Google Fonts) — was selected for all typographic roles, including display headings, body text, and accents. This decision was deliberate and is justified by the following rationale:

  • Geometric sans-serif: Poppins is a geometric typeface with rounded letterforms that conveys a modern, friendly, and approachable personality — appropriate for a tourism brand targeting international and local travellers.
  • Weight range: Poppins supports weights 300–700, providing the full spectrum needed for visual hierarchy (300 for captions, 400 for body, 500–600 for UI labels, 700 for headings and prices).
  • Multilingual support: Poppins supports Latin Extended characters, suitable for displaying international destination names.
  • Web performance: A single font family eliminates multiple font file HTTP requests, reducing Time-to-First-Byte for the typography asset.
Three CSS custom properties (--font-display, --font-body, --font-accent) are all mapped to 'Poppins', sans-serif, allowing per-role font overrides in future iterations without touching component code.

2.2.4 Spacing and Layout System

The layout was built on a consistent spacing and radius token system defined in index.css:

  • Border Radius: 4px (sm), 8px (md), 12px (lg), 16px (xl), 9999px (full — used for pill buttons and avatar circles).
  • Shadow Layers: Four elevation levels (sm, md, lg, xl) using rgba(0,0,0,0.08) for a soft, premium feel.
  • Container: Max-width 1200px centred with 24px horizontal padding, reducing to 16px on mobile.
  • Section Padding: 80px vertical on desktop, reducing to 56px on mobile screens below 640px.

2.2.5 Component Design Principles

All reusable UI components — buttons, badges, form inputs, card components — were implemented as global utility classes in index.css with clearly named variants:

  • Buttons: .btn-primary, .btn-secondary, .btn-outline, .btn-outline-white with size modifiers .btn-sm and .btn-lg. All buttons animate with a 1px translateY lift and box-shadow on hover for tactile feedback.
  • Badges: Six semantic variants (primary, secondary, success, warning, danger, neutral) using transparent background tints of their respective semantic colours.
  • Form Elements: Unified .form-input, .form-select, .form-textarea with a 1.5px solid border that changes colour to the primary blue on focus, accompanied by a 3px spread shadow ring — a common pattern from modern design systems (Material, Ant Design) that provides clear focus visibility for accessibility.

3.1.2 Application Architecture

The React application follows a Layout + Outlet pattern using React Router v6 nested routes. A shared Layout component wraps all public and authenticated pages, rendering the fixed Navbar and Footer. Authentication pages (Login, Register) are deliberately placed outside the Layout route, giving them a distraction-free full-screen design appropriate for onboarding flows.

Global application state (authenticated user, JWT token) is managed via React Context API (AuthContext), which wraps the entire application. The context exposes user, token, isAuthenticated, login(), and logout() to all components without prop drilling. Token persistence is handled via localStorage, with an automatic GET /api/auth/me validation on application load to refresh user state from the server.

3.1.4 Responsive Design Implementation

The application is fully responsive across all device sizes. Three CSS breakpoints are applied consistently:

  • ≤ 1024px: 3-column grids transition to 2-column. The About page story layout changes from side-by-side to stacked.
  • ≤ 768px: Desktop navigation is hidden (display: none). The hamburger menu button becomes visible. The authenticated user menu in the navbar is replaced by equivalent options inside the mobile menu drawer.
  • ≤ 640px: All grids collapse to a single column. Section padding reduces from 80px to 56px. Container horizontal padding reduces from 24px to 16px.

3.1.5 Route Protection

Route protection is implemented via a ProtectedRoute higher-order component that wraps the /dashboard route. It reads the authentication state from AuthContext. If loading is true (initial token validation in progress), a full-page spinner is displayed to prevent a flash of unauthenticated content. If isAuthenticated is false, the user is redirected to /login with the original location preserved in router state. Additionally, an allowedRoles prop enables future fine-grained role-based route protection.

3.2.2 Application Architecture

The backend follows an MVC-inspired layered architecture with clear separation of concerns:

  • Routes (src/routes/): Define HTTP method + path mappings and apply route-level middleware (authentication, role checks). Four route files: authRoutes.js, packageRoutes.js, bookingRoutes.js, inquiryRoutes.js.
  • Controllers (src/controllers/): Handle business logic, interact with the database via parameterised queries, and return structured JSON responses. Each controller corresponds to one route file.
  • Middleware (src/middleware/): authMiddleware.js validates the JWT bearer token on each protected request; errorMiddleware.js provides a centralised 404 and 500 error handler that returns consistent { success: false, message } response envelopes.
  • Config (src/config/): initDb.js establishes the PostgreSQL connection pool on startup and creates all required tables if they do not exist (CREATE TABLE IF NOT EXISTS), enabling a zero-migration fresh deployment workflow.
  • Utils (src/utils/): Shared helper functions (token generation, response formatters).

3.2.4 Authentication and Security

The authentication system implements industry-standard security practices throughout:

  • Password hashing: All passwords are hashed using bcryptjs before storage. The algorithm's adaptive cost factor ensures password hashing remains computationally expensive as hardware improves, protecting against brute-force attacks on any future data breach.
  • JWT issuance and validation: On successful login or registration, the server signs a JSON Web Token containing the user's ID, email, and role as the payload. The token is signed with a secret key stored in the .env file. Subsequent protected requests are validated by decoding and verifying the token in authMiddleware.js, where the req.user object is populated for downstream controller access.
  • Role-based access control (RBAC): Controller functions check req.user.role before executing privileged operations. For example, GET /api/auth/users returns a 403 Forbidden response if the authenticated user's role is not admin.
  • SQL Injection prevention: All database queries use parameterised statements via the pg driver's $1, $2, ... placeholder syntax, which prevents user-supplied input from being interpreted as SQL commands.
  • CORS: The CORS middleware uses a dynamic origin validator that only approves requests from localhost addresses, preventing cross-origin access from production domains during development and from untrusted third-party websites.

3.2.5 Error Handling

A two-layer error handling strategy is employed. At the route level, controllers use try/catch blocks with next(err) to pass errors to the centralised middleware. The notFound middleware catches any unmatched route and creates a descriptive 404 error. The errorHandler middleware normalises all errors into a consistent JSON response envelope: 

{
  "success": false,
  "message": "Human-readable error description",
  "stack": "..." // only included in NODE_ENV=development
}

This design ensures the frontend can reliably check err.response?.data?.message and display it in a toast notification, giving users meaningful error feedback for scenarios such as duplicate email registration or attempting to book a fully-booked tour.

4.4.4 Critical Analysis and Lessons Learned

The UAT results confirm that the core objectives of GlobeTrek Adventures were met to a high standard. The average satisfaction score of 4.6 / 5.0 across all three user types demonstrates that the platform is functionally complete, visually appealing, and intuitive to use across different roles.

Several lessons were learned during the development and testing cycle:

  1. Mobile-first testing should be continuous, not final: Several minor CSS layout issues (icon alignment in mobile menu, textarea scroll in the inquiry response form) were only discovered during the final responsive testing phase. Integrating mobile viewpoint checks at each development iteration would catch these earlier.
  2. Role-based UI architecture is a design decision, not just a code decision: The choice to use a single /dashboard URL with role-adaptive tab rendering (rather than separate routes per role) significantly simplified navigation but required careful conditional rendering logic in the Dashboard component. This was the right choice for this scale of application.
  3. Inline validation feedback dramatically reduces user frustration: Testing showed that users completed the registration form significantly faster with inline error messages (errors appearing on submit) compared to a pattern where errors only appear after a network round-trip. The frontend-first validation implemented in RegisterPage.jsx is validated as the correct UX pattern.
  4. Feature gaps for future iterations: Based on UAT feedback, three features are prioritised for the next development iteration: (a) a price range filter on the Tours page, (b) booking table sorting and pagination in the admin dashboard, and (c) an admin statistics summary panel.
Security Note: During testing, it was confirmed that attempting to access the GET /api/auth/users endpoint directly via a HTTP client (Postman) with a customer-role JWT token returned HTTP 403 Forbidden with the message "Access denied. Admin only." This confirms that the RBAC enforcement operates at the API layer, not just the frontend — ensuring security even if the frontend role checks are bypassed.

5. Conclusion

This report has documented the complete lifecycle of the GlobeTrek Adventures web application — from comparative system analysis and UI design through full-stack implementation to systematic testing and user acceptance evaluation. The platform was developed to directly address the operational needs of a travel and tourism management company, providing distinct, well-defined experiences for three user roles: Customer, Staff, and Administrator.

The comparative analysis of TripAdvisor and MakeMyTrip in Task 01 provided actionable insights that shaped key architectural decisions — particularly the adoption of a flat navigation model, an inline booking widget, and a role-adaptive dashboard. The design system — centred on the navy-and-gold palette and Poppins typography — successfully communicates the brand identity of a premium Sri Lanka travel company while maintaining high contrast ratios and accessibility standards.

The frontend implementation in React 18 with Vite achieves a fast, lightweight production build (323 kB JS bundle) and a fully responsive layout across mobile, tablet, and desktop viewports. The component-based architecture and CSS Modules approach ensure long-term maintainability and scalability. The backend, built on Node.js, Express.js, and PostgreSQL, implements industry-standard security practices including bcrypt password hashing, JWT authentication, parameterised SQL queries, and role-based API access control — demonstrating comprehensive full-stack security awareness.

All 13 defined test cases passed successfully, and User Acceptance Testing returned a mean satisfaction score of 4.6 / 5.0 from three representative users. The testing phase also identified three concrete areas for future enhancement: a price range filter, admin dashboard sorting/pagination, and a statistics summary panel — demonstrating an evidence-driven approach to iterative development.

GlobeTrek Adventures, as delivered, represents a robust, secure, and user-friendly foundation for a real-world travel booking platform — directly fulfilling the functional requirements specified in the CSE5009 WRIT1 assessment brief.
